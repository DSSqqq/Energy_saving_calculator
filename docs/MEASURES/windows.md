# Мероприятие «Окна» — формулы и .docx-выгрузка

Slug: **`windows`**. Полное название: «Улучшение теплозащитных свойств оконных блоков».

Источник методики: типовой раздел энерго-аудита (пример — Баян-Сулу, раздел 3.3.6).

## Обозначения

| Символ | Описание | Ед. изм. |
|--------|----------|----------|
| `S` | Площадь проблемных участков остекления | м² |
| `z` | Продолжительность отопительного периода | сут. |
| `t_в` | Внутренняя температура | °C |
| `t_н` | Средняя t наружного воздуха за `z` | °C |
| `ΔT` | `t_в − t_н` | °C |
| `R_до`, `R_после` | Сопротивление теплопередаче | м²·°C/Вт |
| `g_inf` | Коэф. воздухопроницаемости | кг/(м²·ч) |
| `c` | Удельная теплоёмкость воздуха | ккал/(кг·°С) |
| `k` | Коэф. влияния встречного теплового потока | — |
| `a` | Теплотворная способность газа | Гкал/тыс. м³ |
| `Cээ` | Тариф на газ | тг/м³ |

## Формулы

Per-building (на каждое здание):

```text
ΔT     = t_в − t_н
Q_т    = S · ΔT · z · 24 · (1/R_до − 1/R_после) / 10^6        [Гкал/год]
q_inf  = 0,28 · c · g_inf · k · ΔT                             [ккал/(м²·ч)]
Q_inf  = (q_до − q_после) · S · z · 24 / 10^6                  [Гкал/год]
Q      = Q_т + Q_inf
```

По проекту:

```text
ΣQ
V_газа = ΣQ / a                       [тыс. м³]
Cэк    = V_газа · 1000 · Cээ          [тг/год]
Z      = Σ (цена_строки · S_total)    [тг]   (сметa)
PBP    = Z / Cэк                      [лет]
NPV/IRR/DPI/DPBP — финансовая модель (постоянный CF, см. ниже).
```

Денежная экономия по зданию — пропорционально его доле в `ΣQ`.

### Пресеты типа здания

| Тип | `z`, сут. | `t_н`, °C |
|-----|----------:|----------:|
| `public` (общественное) | 204 | -7,1 |
| `industrial` (производственное) | 158 | -10,0 |
| `other` (прочее) | 180 | -8,0 |

Эти значения подставляются в форму при выборе типа в UI; пользователь может переопределить вручную.

## Финансовая модель

Реализована в [backend/measures/finance/npv.py](../../backend/measures/finance/npv.py) без внешних зависимостей.

Модель: `−investment` в год 0 + постоянный годовой `CF = Cэк` в годах 1..T.

| Показатель | Формула / метод |
|------------|-----------------|
| `NPV` | `−Z + Σ CF/(1+r)^t`, t=1..T |
| `DPI` | `(NPV + Z) / Z` |
| `IRR` | бисекция по `r` для уравнения `NPV(r) = 0` |
| `PBP` | `Z / CF` |
| `DPBP` | первый год, в котором кумулятивный DCF ≥ Z, с линейной интерполяцией внутри года |
| `npv_by_year` | ряд кумулятивной дисконтированной NPV для графика |

## Что попадает в `.docx`

Сборщик: [backend/measures/windows/docx_export.py](../../backend/measures/windows/docx_export.py). Порядок секций повторяет шаблон 3.3.6:

1. Подпись + **Таблица 3.3.6.1** — площади проблемных участков (строка на здание + ИТОГО).
2. Поясняющие абзацы.
3. Подпись + **Таблица 3.3.6.2** — исходные данные (общие константы + per-building строки «Период…», «Средняя t…»).
4. **На каждое здание**: блок Q_т (исходные данные → формула → подстановка чисел → результат в Гкал).
5. **На каждое здание**: блок Q_inf (константы → q_до и q_после → объём сэкономленной энергии).
6. Сводка по зданиям, ΣQ, перевод в тыс. м³ газа, экономия в тг, инвестиции, простой PBP.
7. Подпись + **Таблица 3.3.6.3** — материалы и стоимость (по введённой смете).
8. Подпись + **Таблица 3.3.6.4** — экономия по зданиям (площадь, Гкал, тг + ИТОГО).
9. Подпись + **Таблица 3.3.1.1** — финансовые показатели (ставка, инвестиции, NPV, DPI, IRR, PBP, DPBP).
10. Подпись + **Диаграмма 3.3.1.1** — кумулятивная дисконтированная NPV (PNG из matplotlib).

## Шаблон Word-вёрстки

Чтобы итоговый `.docx` визуально совпадал с исходником клиента, выгрузка построена на **базовом шаблоне**:

- [backend/measures/windows/template/base.docx](../../backend/measures/windows/template/base.docx) — это исходный `.docx` с обнулённым телом, удалёнными картинками и чартами; остаются только `styles.xml`, `theme1.xml`, `settings.xml`, `fontTable.xml`, `numbering.xml`. `python-docx` открывает базу, и все добавляемые программно абзацы и таблицы наследуют тему/шрифты/размеры исходника.
- Чтобы пересобрать базу из обновлённого шаблона клиента, запустите:

```powershell
cd backend
.\.venv\Scripts\python.exe scripts\strip_base.py "c:\path\to\new_source.docx"
```

(Скрипт [backend/scripts/strip_base.py](../../backend/scripts/strip_base.py) — одноразовый, его не нужно запускать на каждый экспорт.)

- Стилевые хелперы — в [backend/measures/docx_utils/builder.py](../../backend/measures/docx_utils/builder.py) (красная строка 1,25 см, выравнивание по ширине, центровка таблицы, узкие поля ячеек, сплошные границы). Для таблиц предусмотрено принудительное отключение автоподбора (`autofit = False`) и явное задание ширины колонок (`col_widths`), что предотвращает их "скачки" при заполнении данными.
- Математика — в [backend/measures/windows/equations.py](../../backend/measures/windows/equations.py): мини-DSL (`m_run`, `m_frac`, `m_sub`, `m_sup`, `m_paren`, `m_para`) и сборщики `eq_q_t`, `eq_q_inf_specific`, `eq_q_infiltration`, `eq_v_gas`, `eq_money`, `eq_pbp`, `eq_sum_two`. Каждая формула — настоящий Word-объект OMML на Cambria Math italic; в `.docx` это полноценные редактируемые уравнения.

### Особенности форматирования при экспорте (кастомная вёрстка)

Для точного соответствия ручному шаблону в коде экспорта применены специфические правила форматирования:
- **Подписи таблиц и графиков:** Вставляются не через стандартный стиль, а формируются программно как обычный абзац с явным заданием шрифта (`Times New Roman`, `12pt`, `Курсив`, без жирного начертания).
- **Таблицы:** В функциях экспорта (`_section_table_*`) явно задаются ширины каждой колонки через `col_widths=[Cm(...), ...]` для обеспечения единообразного вида. Шапкам таблиц программно задается цвет фона (`D9E2F3`) через прямое редактирование XML-свойств ячейки (`_set_cell_background`).
- **Формулы (OMML):**
  - Длинные формулы (например, `Q_т` и `Q_inf`) разбиваются на несколько строк с помощью структуры матрицы/массива уравнений `<m:eqArr>` (функция `m_eqArr` в `equations.py`), чтобы они не выходили за поля страницы.
  - При вставке формул в документ (`_add_equation`) они выравниваются по центру за счет обнуления отступов абзаца и добавления тега `<m:jc m:val="centerGroup"/>` непосредственно в свойства самой OMML-формулы (`<m:oMathParaPr>`).
- **Отступы и разрывы:** Для строгого контроля расстояний между элементами (текстом, формулами, таблицами) используются принудительные пустые абзацы (`doc.add_paragraph()`), а также явный разрыв страницы (`doc.add_page_break()`) перед крупными таблицами.
- **График NPV:** График (`chart.py`) строится с использованием шрифта `Times New Roman`, строгими шагами по осям (`MultipleLocator` — 1 год по оси X и 100 000 по оси Y), плотной сеткой и легендой под графиком. При сохранении применяется `bbox_inches="tight"`, чтобы легенда не обрезалась.
- **Ориентация страницы с графиком:** Перед вставкой графика в документ программно добавляется разрыв раздела, и новая страница переводится в **альбомную ориентацию (Landscape)**. При этом явно меняются местами ширина и высота страницы. За счет альбомной ориентации размер самого графика увеличен (ширина 9 дюймов).

## Известные приближения

- Числа примера из шаблона (27,49 Гкал, 156 280 тг, NPV 384 045 тг и т.п.) могут не совпадать до копейки: приложение считает по приведённым формулам и текущим значениям тарифа/теплотворной — расхождения с шаблоном объясняются интерпретацией тарифа и порядком округлений в исходном документе.
- Доля воздухопроницаемости `0,28` в `q_inf` — переводной коэффициент, обычно 0,28 ккал·с/(м²·°С·кг·ч); вынесен жёстко в `engine.py`. При уточнении методики (если нужны другие коэффициенты) меняйте `_q_specific_inf_kcal_per_m2_h` в `backend/measures/windows/engine.py`.
- В очень старых версиях Word (≤ 2007) объекты OMML отрисовываются хуже; современные (2010+) рендерят их как родные.
